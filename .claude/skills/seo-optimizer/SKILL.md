# SEO Optimizer

Сделать сайт находимым и хорошо представленным в поиске — **сразу для Яндекса и
Google** — через аудит того, что уже есть, автоматическое применение безопасных
правок из кода и понятный план для того, что требует человека (аккаунты,
решения по контенту, доступ к серверу).

## Почему такое разделение

Большая часть SEO-выигрыша — механическая и живёт в коде: sitemap, `robots.txt`
со ссылкой на него, микроразметка, корректные теги `<head>`, canonical-адреса.
Это нужно просто **делать** — детерминированно и низкорисково. Меньшая часть
требует человека: создать аккаунт в Яндекс.Вебмастере / Google Search Console,
выбрать целевые запросы, переписать «тонкий» текст, настроить HTTPS/редиректы на
хостинге. Для этого — **точно объяснить, что сделать**, а не угадывать.

Цель — сайт, который поисковики могут полностью обойти, верно понять и богато
показать (адресный сниппет, FAQ-сниппет, фавикон, хороший title/description).

## Рабочий процесс

Идти по фазам по порядку. Вести их как todo, чтобы ничего не потерять.

### 1. Разведка (Discover)

Понять сайт до того, как что-то менять:

- Найти страницы: `*.html` в корне или каталог маршрутов/страниц фреймворка.
- Определить стек (см. `references/stacks.md`): статический HTML,
  React/Next/Vue, WordPress и т.д. От этого зависит, *куда* вносить правки.
- **Проверить, рукописный HTML или генерируемый** (критично — см. раздел
  «ГЛАВНОЕ перед любой правкой» в `references/stacks.md`). Признаки генерации:
  маркеры `<!-- ADMIN:… -->`/шаблонные вставки, файл данных (`content.json` и
  т.п.), скрипт сборки (`build.js`), кастомная админка в репозитории. Если HTML
  собирается из данных — SEO вставлять **в источник** (шаблон/рендер-функцию/
  data-файл), иначе правки в готовом HTML затрёт при первой же пересборке.
- Прочитать существующий `<head>` каждой страницы, а также `robots.txt`,
  `sitemap.xml`, `.htaccess`, манифест и текущие теги аналитики/подтверждения.
- Установить канонический домен и протокол (https + с www или без). Если из
  конфигов неочевидно — спросить пользователя: каждая правка зависит от знания
  реального URL.

### 2. Аудит (Audit)

Запустить встроенный аудитор для объективного повторяемого снимка:

```bash
python <skill-dir>/scripts/audit.py <путь-к-сайту-или-html-файлам...>
```

Он отчитывается по каждой странице, что есть и чего нет по всему чек-листу
(title, description, h1, canonical, OG, Twitter, JSON-LD, viewport, lang,
покрытие alt, фавикон) и по сайту в целом (robots.txt, sitemap.xml). Использовать
его JSON как основу отчёта — не проверять глазами то, что скрипт проверяет
надёжно. Поверх — добавить суждения, которые скрипт сделать не может (релевантен
ли title запросам? достаточно ли индексируемого текста? не «тонкий» ли контент?),
опираясь на `references/checklist.md`.

### 3. План (Plan)

Сформировать приоритизированный отчёт строго по структуре из раздела
**«Формат отчёта»** ниже. Каждый пункт помечен **[АВТО]** (применишь сейчас сам)
или **[ВРУЧНУЮ]** (нужен человек — дать пошаговую инструкцию). Сортировать по
влиянию: сначала блокеры индексации и микроразметка под сниппеты, косметика — в
конце.

### 4. Применить правки [АВТО]

Внести прямо в код (полные шаблоны — в `references/schema-templates.md`,
размещение под стек — в `references/stacks.md`):

- **`sitemap.xml`** — сгенерировать из списка страниц:
  `python <skill-dir>/scripts/gen_sitemap.py --base https://domain <страницы...>`
- **`robots.txt`** — разрешить обход, оставить CSS/JS открытыми (ботам они нужны
  для рендера), добавить строку `Sitemap:`.
- **Микроразметка (JSON-LD)** — `Organization`/`LocalBusiness` для
  контактов+адреса (даёт адресный сниппет в Яндексе), `FAQPage`, если есть FAQ,
  `BreadcrumbList` для многостраничников. Обычно это самая ценная правка и она
  невидима для дизайна.
- **Основа `<head>`** — `<title>`, `<meta name="description">`, `canonical`,
  `viewport`, `lang`, теги Open Graph + Twitter Card, ссылки на фавикон.
- **По картинкам** — осмысленный `alt`; помечать «текст-картинкой».

Уважать правила проекта (например, гайд в CLAUDE.md). Всё перечисленное невидимо
для вёрстки, поэтому не конфликтует с дизайн-системой — но никогда не
перекрашивать видимый UI ради SEO без явной просьбы.

После правок **перезапустить `audit.py`** и убедиться, что пункты [АВТО] теперь
проходят. Показать доказательства, а не просто заявить об успехе.

### 5. Передать пункты [ВРУЧНУЮ]

По каждому дать конкретные шаги (см. `references/manual-steps.md`): подтверждение
прав в Вебмастере / Search Console, отправка sitemap, установка Метрики/GA,
подбор запросов через Wordstat, настройка HTTPS/www-редиректов на хостинге,
исправление «тонкого» контента. Конкретно — назвать точный инструмент и точное
поле.

### 6. Проверка на проде и итог

«Закоммитил» ≠ «работает»: после деплоя проверь на **живом** домене, что правки
реально доехали и хостинг их не игнорирует (см. `references/deploy-and-prod.md`).
Минимум через `curl`: коды ответа, редиректы (http→https, www→без-www,
index.html→/), и что `canonical`/OG/JSON-LD/`<title>` реально в отданном HTML
(особенно на CMS/SPA). Частая причина «правки не сработали» — деплой не скопировал
дотфайлы (`.htaccess`) или хостинг за прокси. Затем перечислить, что изменилось,
что осталось на человека, и предложить закоммитить/задеплоить.

Если речь зашла об индексации — помнить: это график поисковика, а не кнопка
(«обойдено 200, но нет в базе» = норма). Ускорители и сроки — в
`references/deploy-and-prod.md`.

## Формат отчёта

Использовать эту структуру, чтобы вывод был сканируемым и пользователь точно
знал, что сделано, а что — на нём:

```
# SEO-аудит — <домен>

## Что уже хорошо ✅
- <пункт> — <почему важно>

## Делаю сейчас (автоматически)
🔴 Высокий
- [АВТО] <правка> — <одна строка зачем / какому поисковику помогает>
🟡 Средний
- [АВТО] <правка> — <зачем>

## Нужно от тебя (вручную)
🔴 Высокий
- [ВРУЧНУЮ] <задача> — <пошагово или ссылка на точный инструмент>

## Итог
- Применено: <список>.  Осталось вручную: <список>.
```

«Зачем» — по одной строке: пользователю нужен сигнал, а не учебник. Где уместно,
привязывать пункт к поисковику («даёт адресный сниппет в Яндексе», «Core Web
Vitals у Google»).

## Файлы-справочники

Читать по мере необходимости — не вставлять их содержимое заранее:

- `references/checklist.md` — полный чек-лист аудита Яндекс + Google,
  сгруппированный и приоритизированный, с правилом и способом проверки каждого
  пункта. Источник истины.
- `references/schema-templates.md` — готовый к копированию JSON-LD (Organization,
  LocalBusiness, FAQPage, BreadcrumbList, Product, Article, WebSite) плюс
  шаблоны head / OG / robots / sitemap с плейсхолдерами.
- `references/stacks.md` — куда идёт каждая правка под стек (статический HTML,
  Next.js, Vite/React SPA, WordPress) и нюансы стека (пре-рендер SPA, Metadata
  API в Next, плагины WP).
- `references/manual-steps.md` — задачи только для человека с точными
  инструкциями: настройка Вебмастера и Search Console, отправка sitemap,
  Метрика/GA, подбор запросов в Wordstat, настройка HTTPS/редиректов.
- `references/deploy-and-prod.md` — грабли деплоя и хостинга (дотфайлы/`.htaccess`
  не доезжают, прокси-SSL, дубль `/index.html`), готовый блок редиректов и
  обязательная проверка на проде через `curl`. Плюс: почему индексация — это
  график поисковика, а не кнопка, и чем её ускорить (sitemap, переобход, обход по
  счётчику аналитики, IndexNow). Читать на фазах «Проверка на проде» и при любых
  вопросах «правки не сработали» / «страница не в поиске».

## Скрипты

- `scripts/audit.py` — сканирует HTML и файлы сайта, печатает отчёт «есть/нет» по
  каждой странице и по сайту в JSON (и читаемую сводку). Детерминированный;
  использовать для аудита и повторно — после правок, чтобы доказать, что они
  применились.
- `scripts/gen_sitemap.py` — собирает валидный `sitemap.xml` из базового URL и
  списка страниц/путей.

Оба — на Python 3 без зависимостей (только стандартная библиотека).
