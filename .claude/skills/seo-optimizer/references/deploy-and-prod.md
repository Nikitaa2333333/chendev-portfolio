# Деплой и проверка на проде

SEO-правки в коде бесполезны, если они не доехали на сервер или хостинг их
игнорирует. Поэтому после правок — всегда проверяй на **живом** сайте, а не
только локально. Ниже — провайдер-нейтральные грабли и проверки (примеры
хостингов — лишь иллюстрация, принцип общий).

## Грабли деплоя и хостинга

### 1. Деплой теряет дотфайлы (`.htaccess` и пр.)
Скрипты вида `cp -r src/* dst` и `rm -rf dst/*` **не трогают дотфайлы** — glob
`*` их не видит. Итог: `.htaccess` (а с ним редиректы, кэш, правила) никогда не
попадает на сервер, хотя в репозитории лежит.
- Чинить: `cp -r src/. dst` (копирует и скрытые), `shopt -s dotglob`, либо
  `rsync` (копирует дотфайлы по умолчанию). В CI — проверь шаг копирования.
- Симптом: правила из `.htaccess` «не работают», хотя файл закоммичен.

### 2. Хостинг за прокси: SSL терминируется не на Apache
На многих shared-хостингах перед Apache стоит прокси/nginx, который и держит
SSL. Тогда:
- `RewriteCond %{HTTPS} off` **ненадёжно** (Apache видит http от прокси) —
  редирект на https либо не срабатывает, либо зацикливается. Используй
  `RewriteCond %{HTTP:X-Forwarded-Proto} !https`.
- Статику часто отдаёт сам прокси, **минуя `.htaccess`** → твои `Cache-Control`
  для картинок/шрифтов игнорируются. Кэш/сжатие тогда настраивается на стороне
  хостинга, а не в `.htaccess`.
- Форс-HTTPS и «главное зеркало» на таких хостингах часто проще включить
  **переключателем в панели**, чем правилом в `.htaccess`.

### 3. Дубль главной по `/index.html` (характерно для статики/Apache)
Сервер отдаёт главную и на `/`, и на `/index.html` (DirectoryIndex). На Next/SPA/
CMS такого дубля обычно нет — там роутер сам отдаёт один URL. Если
внутренние ссылки ведут на `index.html`, поисковик проиндексирует именно этот
«некрасивый» URL.
- Чинить: 301 `/index.html` → `/` **(абсолютным https, иначе за прокси будет
  лишний http-хоп)** + все внутренние ссылки указывают на `/`.

### Готовый блок `.htaccess` (вариант для статики/Apache)
⚠️ Это реализация **для статики на Apache**. На других стеках те же редиректы
делаются своими средствами: **Nginx** — `return 301` в конфиге; **Next/Nuxt** —
`redirects()` / middleware; **CMS** — плагин или настройки; **любой хостинг** —
переключатель в панели. Принцип один (единый https-адрес без www и без дублей),
способ — по стеку (см. stacks.md). Домен-нейтрально через `%{HTTP_HOST}`:
```apache
<IfModule mod_rewrite.c>
  RewriteEngine On

  # http -> https (за прокси протокол приходит в X-Forwarded-Proto)
  RewriteCond %{HTTP:X-Forwarded-Proto} !https
  RewriteRule ^ https://%{HTTP_HOST}%{REQUEST_URI} [L,R=301,NE]

  # www -> без www
  RewriteCond %{HTTP_HOST} ^www\.(.+)$ [NC]
  RewriteRule ^ https://%1%{REQUEST_URI} [L,R=301,NE]

  # /index.html -> / (убрать дубль главной)
  RewriteCond %{THE_REQUEST} \s/+index\.html[\s?] [NC]
  RewriteRule ^index\.html$ https://%{HTTP_HOST}/ [L,R=301,NE]
</IfModule>
```

## Проверка на проде после деплоя (обязательный шаг)
«Закоммитил» ≠ «работает». Прогони на ЖИВОМ домене (подставь свой):
```bash
# Коды ответа и редиректы (-I = только заголовки, без -L = не следовать)
curl -sI http://site/            # ожидаем 301 -> https
curl -sI https://www.site/       # ожидаем 301 -> без www
curl -sI https://site/index.html # ожидаем 301 -> https://site/
curl -sI https://site/           # ожидаем 200

# canonical / OG / JSON-LD / title РЕАЛЬНО в отданном HTML
# (критично для CMS и SPA — там теги могут не дорендериться на сервере)
curl -s https://site/ | grep -iE 'rel="canonical"|og:title|application/ld\+json|<title>'

# Служебные файлы доступны и нужного типа
curl -s -o /dev/null -w '%{http_code}\n' https://site/sitemap.xml
curl -s -o /dev/null -w '%{http_code} %{content_type}\n' https://site/favicon.ico
```
Если редиректы/заголовки не совпали с тем, что в `.htaccess` → почти наверняка
п.1 (дотфайл не доехал) или п.2 (прокси). Сначала чини доставку/хостинг, потом
ищи логику.

## Индексация — это график, а не переключатель
После того как технически всё чисто, попадание в индекс — **решение поисковика
на его цикле обновления**, мгновенной кнопки нет. Нормальный промежуточный
статус: страница обойдена (код 200), но «ещё нет в поисковой базе» — она в
очереди. Не паникуй и не правь код по кругу.

Что реально ускоряет (по убыванию контроля):
- **Sitemap** отправлен в панели вебмастера.
- **Ручной переобход** конкретного URL в панели (Яндекс: «Переобход страниц»;
  Google Search Console: «Проверка URL» → «Запросить индексирование»).
- **Обход по счётчику аналитики** — Яндекс умеет обходить страницы по данным
  Метрики (привязать счётчик к Вебмастеру и включить «Обход по счётчикам»).
  Сильный ускоритель для молодых сайтов.
- **IndexNow** — мгновенный пинг «обойди URL» (поддерживают Яндекс и Bing;
  Google — нет). Требует ключ-файл в корне.
- Срок: у молодого сайта главная заходит обычно за дни — ~2 недели. Внутренние
  страницы часто индексируются раньше главной — это нормально, не баг.

Если индексируется «старый» адрес (например `/index.html`), а не канонический —
поставь 301 на канонический и отправь старый URL на переобход: поисковик увидит
редирект + `canonical` и переклеит индекс.

**Склейка зеркал (http/https/www) у Яндекса — это отдельная заявка, не только
301.** Яндекс считает `http`/`https`/`www`/без-www **разными сайтами**. Чтобы он
быстро выбрал главное зеркало: добавь оба адреса в Вебмастер, настрой 301, затем
Вебмастер → **Индексирование → Переезд сайта** → отправь заявку. Симптом
незавершённой склейки — статус страниц «данные о каноническом адресе
обрабатываются». Подробные шаги — в manual-steps.md.
