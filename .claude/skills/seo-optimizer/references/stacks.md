# Куда вносить правки под разный стек

Определи стек на фазе разведки, затем применяй правки в правильном месте. Сами
правки одинаковы (см. checklist + schema-templates) — отличается *куда* их класть
и подводные камни.

## Как определить стек
- `package.json` с `next` → **Next.js**; с `react`/`vite` без SSR → **SPA**.
- `wp-content/`, `wp-config.php` → **WordPress**.
- Просто `*.html` + ассеты, без сборки → **статический HTML**.
- `nuxt`/`vue`, `astro`, `gatsby`, `sveltekit` → по аналогии (SSR/SSG как Next,
  чистый клиент — как SPA).

---

## ⚠️ ГЛАВНОЕ перед любой правкой: HTML рукописный или генерируемый?

Прежде чем трогать `index.html` (или любой `*.html`), выясни — **этот файл пишут
руками или его генерирует код из данных?** Если генерирует, а ты правишь готовый
HTML — **твои теги затрёт при следующей пересборке/публикации**. Это самая частая
и обидная ошибка: SEO «применилось», прошло мимо ревью, а через день контент-
менеджер нажал «сохранить» в админке — и canonical/OG/JSON-LD исчезли.

**Признаки генерируемого HTML** (ищи на фазе разведки):
- комментарии-маркеры в разметке: `<!-- ADMIN:... start/end -->`, `<!-- BEGIN ... -->`,
  Handlebars/EJS/Jinja-вставки (`{{ }}`, `{% %}`), `<!--build:...-->`;
- файл данных контента: `data/content.json`, `*.yml`, `_data/`, headless-CMS;
- скрипт сборки: `build.js`, `gulpfile`, `vite.config`, генератор статики;
- кастомная админка в репозитории (`admin/`), которая коммитит `index.html`
  (грепни её код на `index.html`, `applyContent`, `innerHTML`, `getText`).

**Правило:** нашёл генератор/шаблон/маркеры — **вставляй SEO в источник**
(функцию рендера `<head>`, шаблон, поля в data-файле), а не в собранный HTML.
Потом запусти сборку и проверь результат `audit.py` по собранному файлу. Так
теги переживают любую публикацию.

**Пример (реальный кейс).** Кастомная static-admin: `admin/` коммитит `index.html`,
который собирается из `data/content.json` функцией `renderHead()` в шаблонном
модуле. Замена контента между `<!-- ADMIN:head start/end -->` стирала вручную
добавленные теги. Решение — добавить генерацию canonical/OG/Twitter/JSON-LD
**внутрь `renderHead()`**, забирая данные (адрес, телефоны, FAQ) из
`content.json`. После этого и админка, и `build.js` всегда восстанавливают SEO.
Доп. нюанс: если админка уже открыта в браузере, она держит устаревшую копию
HTML — предупреди пользователя перезагрузить её перед публикацией.

## Статический HTML (включая Tailwind через CDN)

Самый простой случай — правки идут напрямую.
- `<head>`-теги, JSON-LD — прямо в каждый `*.html`. Если страниц много и шапка
  повторяется, всё равно правим в каждом файле (нет шаблонизатора) — будь
  внимателен к уникальности title/description/canonical на каждой.
- `robots.txt`, `sitemap.xml`, фавиконы — в корень сайта.
- Редиректы https/www, кэш, 404 — `.htaccess` (Apache) или конфиг хостинга
  (см. manual-steps.md).
- Подводный камень: легко забыть продублировать аналитику и canonical на
  второстепенных страницах — проверь грепом по всем `*.html`.

## Next.js (App Router)

- Мета — через **Metadata API**, не руками в `<head>`. В `layout.tsx`/`page.tsx`:
  ```ts
  export const metadata = {
    title: "...", description: "...",
    alternates: { canonical: "https://домен/путь" },
    openGraph: { title, description, url, images: ["..."] },
  }
  ```
- JSON-LD — отрендерить `<script type="application/ld+json"
  dangerouslySetInnerHTML={{__html: JSON.stringify(data)}} />` в серверном
  компоненте (чтобы попало в HTML на сервере).
- `sitemap.xml` и `robots.txt` — через `app/sitemap.ts` и `app/robots.ts`
  (нативная генерация Next), не статические файлы.
- Pages Router — мета через `next/head`, sitemap обычно сторонним пакетом.
- Подводный камень: убедись, что данные в HTML от сервера (SSR/SSG), а не
  доклеиваются на клиенте.

## SPA (Vite/CRA/Vue без SSR)

Главная проблема SEO у SPA: HTML почти пустой, контент рисует JS. Google рендерит
JS давно. **Яндекс тоже умеет рендерить JS** (бета, по умолчанию «на усмотрение
робота»), но менее надёжно, и ссылки на JS-навигации без `<a href>` не обходит.
- Минимум: заполнить статический `index.html` (title, description, OG, JSON-LD
  организации) — это попадёт в исходный HTML.
- Динамические мета по роутам — `react-helmet-async`/`@unhead/vue`, но это всё
  равно клиентский рендер.
- Управление рендерингом у Яндекса: Вебмастер → **Индексирование → Рендеринг
  страниц JavaScript** (тумблер). Если есть SSR/пре-рендер — рендеринг роботом
  лучше **выключить** (отдаёшь готовый HTML сам). Для контента с задержкой —
  `window.YandexRotorSettings` (сообщить роботу, что страница догрузилась).
- 🔴 **Рекомендация для серьёзного SEO**: не полагаться на JS-рендеринг робота, а
  делать пре-рендер (`vite-plugin-ssr`, `prerender`, `react-snap`) или SSR/SSG
  (Next/Nuxt/Astro) — тогда и Яндекс, и Google гарантированно видят контент. Без
  этого Яндекс может видеть пустую страницу — отметь как **[ВРУЧНУЮ]** решение.
- Навигация — настоящие `<a href>` (роутер их перехватывает), не `onClick` по
  `<div>`.

## WordPress

- Мета/OG/canonical/sitemap/schema — обычно через плагин **Yoast SEO** или **Rank
  Math**. Не дублируй вручную то, что делает плагин (конфликт двойных тегов).
  - Проверь, установлен ли плагин; если да — настраивай в нём, не в шаблоне.
  - Если плагина нет — поставить его проще, чем править `header.php`.
- JSON-LD: Yoast/Rank Math генерируют Organization/Breadcrumb автоматически —
  проверь вывод, не плодя дубли.
- `robots.txt` и sitemap часто отдаёт сам плагин (`/sitemap_index.xml`).
- Подводный камень: тема может хардкодить свой `<title>`/мусорные теги — ищи
  дубли. Кэш-плагины — чистить после правок.

## Общий принцип

Если есть шаблонизатор/сборка — правь источник (layout/шаблон/конфиг), а не
сгенерированный HTML. Для чистой статики — каждый файл напрямую. Всегда проверяй
итог в собранном HTML (`audit.py` по билду, а не по исходникам, если они
различаются).
